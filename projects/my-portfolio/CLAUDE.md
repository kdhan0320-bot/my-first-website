# 김도한 포트폴리오 작업 규칙

세부 디자인 스펙(색상 hex, px, 초 단위 타이밍, 컴포넌트별 정의, 회차별
변경 이력)은 이 문서가 아니라 `docs/design-reference.md`를 참조한다.

## 목적
이 프로젝트는 김도한의 취업용 웹 포트폴리오다. 목표는 화려한 개인
홈페이지가 아니라, 채용자가 30초 안에 지원 방향, 작업 역량, 대표
프로젝트를 이해할 수 있는 신입 UX/UI·웹퍼블리싱 포트폴리오를 만드는
것이다.

## 지원 방향
- 신입 UX/UI 웹디자이너
- 웹퍼블리셔
- 서비스기획형 프론트엔드 주니어
- AI 활용 UX/UI 포트폴리오
- B2B/업무툴 UI 구현형 주니어

## 핵심 포지셔닝
사용자 흐름과 정보 구조를 정리하고, Figma 설계와 React/HTML/CSS/
JavaScript 구현을 연결하는 신입 지원자로 보이게 한다. 순수 그래픽
감각만 강조하지 말고, 정보구조, 사용 흐름, 업무형 화면, 구현 가능성,
AI 보조 활용을 함께 보여준다.

## 작업 판단 기준
항상 아래 순서로 판단한다.
1. 취업 포트폴리오 완성도
2. 채용자가 빠르게 이해할 수 있는 구조
3. 실제 사용성
4. 반응형/접근성/링크 신뢰도
5. 최신 UI/UX 흐름 반영
6. 사용자가 지속할 수 있는 작업 범위

화려함보다 명확성, 과한 기능보다 신뢰성, 새 프로젝트보다 기존 프로젝트
완성을 우선한다.

## 디자인 방향 핵심 원칙
콘셉트명은 "Ordered Signal"(상세는 `docs/design-reference.md`). Figma 파일
`53Ppn2hIgrvs9Jra3eejFs`("ORDERED SIGNAL / STYLE GUIDE")를 기준으로 이전
"Dohan.K Flow Blueprint" 시스템을 전면 교체했다(사용자 승인 완료).

- Ink Black/Deep Slate(다크) + Warm Ivory(라이트) 배경이 섹션별로
  교차하고, Signal Orange를 포인트 컬러로 쓰는 하이컨트라스트 에디토리얼
  톤. 색상 6종 전체 정의와 사용 규칙은 `docs/design-reference.md` 참조.
- IBM Plex Sans KR(한글 제목/본문) + IBM Plex Mono(라벨/번호/메타데이터)
  2종 폰트만 쓴다.
- 큰 한글 타이포 1개가 화면을 주도하고, 포인트 컬러는 전체 면적의
  5~8% 이내로 제한한다.
- 금지: 3D·파티클·글리치·강한 blur·마우스 추적·자동 무한 marquee.
  진입 시 1회 fade/align 모션은 허용하되 무한 반복 애니메이션은 두지
  않는다(Hero/Contact 포함, 예외 없음).
- 모션은 장식이 아니라 "정리→정렬→완성" 흐름을 보여주는 용도로만 쓴다.
  Hero는 Scattered→Align→Final 3단계, Contact는 짧은 신호 정렬 모션 —
  둘 다 1회 재생, `prefers-reduced-motion`에서는 Final 상태 즉시 표시.
- 전체 리디자인은 위 Figma 승인 건으로 이미 반영이 끝났으므로, 이후
  추가 요청은 다시 "사용자가 명시한 섹션 범위 내 조정"부터 계획을
  제안하고 진행한다(임의 재변경 금지).
- 헤더 nav는 PROJECTS / GITHUB / PDF PORTFOLIO(있을 때만) / 메일 CTA
  버튼 구성을 기본으로 유지한다.

## 외부 레퍼런스 활용 원칙
- 특정 사이트를 그대로 복제하지 않는다.
- 레이아웃/컴포넌트를 그대로 베끼지 않는다.
- 디자인 기준은 Figma 파일과 `docs/design-reference.md`를 우선한다.

## 문장 규칙
- 거짓말, 과장, 억지 칭찬, 검증되지 않은 성과 수치 금지
- 실제 API가 없으면 "데모", "프로토타입", "예시 데이터 기반"으로 표시
- "실무 수준", "완벽", "효율 00% 향상", "사용자 만족도 향상" 같은 검증
  안 된 표현 금지
- 부족한 점과 한계를 숨기지 말고 담백하게 표현한다

## AI 활용 표현 규칙
AI 활용 사실을 숨기거나 축소하지 않는다. "AI를 썼다/안 썼다"보다 "무엇을
직접 판단하고 검증했는가"를 구체적으로 보여준다(루트 `CLAUDE.md`, `docs/design-trends-2026.md`
4번 항목과 동일 기준).

좋은 표현:
- Claude/ChatGPT로 초안을 빠르게 만들고, 무엇을 검토·수정·반려했는지
  구체적으로 밝힌다 (예: "AI가 제안한 A 구조를 써보니 B 문제가 있어서
  C로 바꿨다").
- 아키텍처·기술을 고른 이유와, 고르지 않은 대안을 함께 밝힌다.
- 사용자가 요구사항을 정의하고 최종 판단과 수정 방향을 결정했음을
  분명히 한다.

나쁜 표현:
- AI가 전체 서비스를 제작했다 / AI로 실무 수준 서비스를 완성했다.
- AI가 사용자 데이터를 분석했다 / AI가 자동으로 디자인을 완성했다.
- AI 활용 사실 자체를 숨기거나 "직접 다 만들었다"처럼 표현한다.

## 프로젝트 관리 원칙
프로젝트 리스트는 고정하지 않는다. 대표 프로젝트와 Archive 프로젝트는
사용자의 최신 지시에 따른다. 기본 원칙:
- 모든 프로젝트를 메인에 올리지 않는다. 대표 프로젝트는 3~4개만 강하게
  보여준다.
- 미완성 프로젝트는 대표로 올리지 않는다.
- 실제 API가 없는 프로젝트는 한계를 표시한다.
- Figma 프로젝트에는 GitHub 버튼을 넣지 않는다.
- 오류 링크가 있는 프로젝트는 수정 전까지 대표로 노출하지 않는다.
- 새 프로젝트 추가보다 기존 프로젝트 정리와 마감을 우선한다.
- 카드 요약/상세 페이지/`/projects` 카드/Preview 등 같은 프로젝트를
  설명하는 문구는 서로 모순되지 않게 하나의 사실 기준으로 통일한다.
  구현 여부가 불확실하면 "구현했다"고 단정하지 말고 "구조 설계",
  "데모", "제한적으로 구현" 등으로 정확히 표현한다.
- 대표 프로젝트(현재 JobFlow/버스 도착정보 앱/Portfolio Feedback Hub)는
  고유 상세 URL(`/projects/:slug`, `ProjectDetailPage.jsx`)을 쓰고,
  나머지 Archive 프로젝트는 기존 모달을 쓴다. 새 프로젝트를 대표로
  올릴 때만 이 기준을 다시 판단한다.
- More Works처럼 "실제 조사 후 공개 판단이 필요한" 영역은 데이터에
  명확한 공개 플래그(예: `moreWorksPublished`)로 표시하고, 공개 판단
  전에는 카드/슬롯을 만들지 않는다.

## 코드 작업 규칙
- 여러 파일 수정, 구조 변경, 디자인 방향 변경, 삭제/이동이 포함되면
  먼저 계획을 제안한다. 오타·문장·소규모 CSS 수정처럼 범위가 명확한
  작업은 바로 처리 후 보고한다.
- 새 라이브러리는 사용자가 명시적으로 요청하지 않으면 추가하지 않는다.
- 파일 삭제 전 참조 여부를 확인하고, import 경로가 깨지지 않게
  확인한다.
- 미사용 이미지, 중복 컴포넌트, 백업 파일, placeholder 데이터는 삭제
  전 "파일 경로 / 삭제 근거 / 참조 여부 / 위험도 / 권장 조치"를 표로
  보고한다. 확실하지 않으면 TODO로 남긴다.
- node_modules, dist, .cache, .vite는 분석 대상에서 제외한다.

## 검증 규칙
- 기본: `npm run build`, 콘솔 에러 없음, 주요 링크(GitHub, Live Demo,
  Prototype) 정상, PDF 버튼은 실제 PDF가 있을 때만 노출(버튼뿐 아니라
  "PDF가 없어서 숨긴다"는 안내 문구도 공개 화면에는 넣지 않는다 — 개발/
  QA 관점의 설명이라 방문자에게는 불필요하다).
- UI/반응형 작업일 때만: 390px, 430px, 768px, 820px, 1024px, 1366px,
  1440px에서 버튼 겹침·가로 스크롤 없음을 확인한다.

## 답변 방식 / 사용자 성향 반영
- 불필요한 칭찬, 막연한 응원, 장황한 배경 설명 금지. 부족한 점은
  솔직하게 말한다.
- 모르는 것은 추측하지 말고 [확인 필요]로 표시한다.
- 사용자는 작업 범위가 커지면 마감이 늦어지는 경향이 있으므로, 새
  프로젝트 추가/전체 리디자인/트렌드 기능 추가보다 오류 링크 수정,
  카드 문장 정리, 대표 프로젝트 선별, 반응형 QA, 접근성 개선처럼 작은
  단위 작업을 우선 제안한다.
