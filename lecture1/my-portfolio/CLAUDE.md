# 김도한 포트폴리오 작업 규칙

## 목적
이 프로젝트는 김도한의 취업용 웹 포트폴리오다.

목표는 화려한 개인 홈페이지가 아니라, 채용자가 30초 안에 지원 방향, 작업 역량, 대표 프로젝트를 이해할 수 있는 신입 UX/UI·웹퍼블리싱 포트폴리오를 만드는 것이다.

## 지원 방향
- 신입 UX/UI 웹디자이너
- 웹퍼블리셔
- 서비스기획형 프론트엔드 주니어
- AI 활용 UX/UI 포트폴리오
- B2B/업무툴 UI 구현형 주니어

## 핵심 포지셔닝
사용자 흐름과 정보 구조를 정리하고, Figma 설계와 React/HTML/CSS/JavaScript 구현을 연결하는 신입 지원자로 보이게 한다.

순수 그래픽 감각만 강조하지 말고, 정보구조, 사용 흐름, 업무형 화면, 구현 가능성, AI 보조 활용을 함께 보여준다.

## 작업 판단 기준
항상 아래 순서로 판단한다.

1. 취업 포트폴리오 완성도
2. 채용자가 빠르게 이해할 수 있는 구조
3. 실제 사용성
4. 반응형/접근성/링크 신뢰도
5. 최신 UI/UX 흐름 반영
6. 사용자가 지속할 수 있는 작업 범위

화려함보다 명확성, 과한 기능보다 신뢰성, 새 프로젝트보다 기존 프로젝트 완성을 우선한다.

## 디자인 방향
- 다크 네이비, 블루그레이, 화이트, 실버 계열 중심
- 차분하고 신뢰감 있는 B2B/업무툴형 분위기
- 콘텐츠 중심의 미니멀한 레이아웃
- 카드형 UI, 명확한 정보 계층, 충분한 여백
- 모바일/태블릿/PC 반응형 안정성 우선
- 우주/네온/장식 요소는 과하면 줄이고 가독성을 우선한다
- 프로젝트 썸네일, 버튼, 라벨, 카드 높이는 일관성을 우선한다

## 외부 레퍼런스 활용 원칙
- 특정 사이트를 그대로 복제하지 않는다.
- 사용자가 지정한 레퍼런스(예: if(kakao)25, FEConf, i-AWARD Korea, DAN25 등)는
  톤앤매너·정보 구조·카드 구성·여백·문장 흐름 참고 용도로만 활용한다.
- 레이아웃/컴포넌트를 그대로 베끼지 않고, 이 프로젝트의 다크 네이비 기반
  디자인 시스템 안에서 재해석한다.
- 사용자가 명시적으로 요청한 경우, 기존 구조를 유지하는 범위에서 시각적
  위계와 레이아웃 밀도를 조정할 수 있다. 단, 외부 사이트의 코드를
  그대로 추출하거나 복제하지는 않는다.

## 2025~2026 UI/UX 반영 기준
최신 트렌드는 무조건 넣는 것이 아니라 취업용 포트폴리오에 도움이 될 때만 적용한다.

우선 반영:
- 접근성
- 명확한 정보 구조
- 반응형 완성도
- 짧고 명확한 문장
- 프로젝트별 문제 정의와 역할
- AI 활용 과정의 투명한 설명
- 작은 마이크로 인터랙션
- 콘텐츠 중심의 bento/card layout

주의:
- 과한 3D 금지
- 과한 glassmorphism 금지
- 과한 네온 효과 금지
- 의미 없는 애니메이션 금지
- 트렌드라는 이유만으로 새 기능 추가 금지
- 시각디자인,기능 등이 너무 ai티가 나지 말것

## 문장 규칙
- 거짓말 금지
- 과장 금지
- 억지 칭찬 금지
- 검증되지 않은 성과 수치 금지
- 실제 API가 없으면 "데모", "프로토타입", "예시 데이터 기반"으로 표시
- "실무 수준", "완벽", "효율 00% 향상", "사용자 만족도 향상" 같은 검증 안 된 표현 금지
- 부족한 점과 한계를 숨기지 말고 담백하게 표현한다

## AI 활용 표현 규칙
AI는 결과물을 대신 만든 도구가 아니라 보조 도구로 표현한다.

좋은 표현:
- Claude/ChatGPT를 활용해 문장 정리, 코드 점검, 개선안 도출을 보조받았다.
- AI 제안을 그대로 사용하지 않고 요구사항에 맞게 수정하고 검토했다.
- 사용자가 요구사항을 정의하고 최종 판단과 수정 방향을 결정했다.

나쁜 표현:
- AI가 전체 서비스를 제작했다.
- AI로 실무 수준 서비스를 완성했다.
- AI가 사용자 데이터를 분석했다.
- AI가 자동으로 디자인을 완성했다.

## 프로젝트 관리 원칙
프로젝트 리스트는 고정하지 않는다.
대표 프로젝트와 Archive 프로젝트는 사용자의 최신 지시에 따른다.

다만 기본 원칙은 다음과 같다.
- 모든 프로젝트를 메인에 올리지 않는다.
- 대표 프로젝트는 3~4개만 강하게 보여준다.
- 미완성 프로젝트는 대표로 올리지 않는다.
- 실제 API가 없는 프로젝트는 한계를 표시한다.
- Figma 프로젝트에는 GitHub 버튼을 넣지 않는다.
- 오류 링크가 있는 프로젝트는 수정 전까지 대표로 노출하지 않는다.
- 새 프로젝트 추가보다 기존 프로젝트 정리와 마감을 우선한다.

## 답변 방식
사용자는 객관적이고 정확한 피드백을 선호한다.

따라서:
- 불필요한 칭찬 금지
- 막연한 응원 금지
- 장황한 배경 설명 금지
- 부족한 점은 솔직하게 말하기
- 실행 가능한 작업으로 정리하기
- 모르는 것은 추측하지 말고 [확인 필요]로 표시하기
- 수정 전에는 계획을 먼저 제안하기
- 수정 후에는 변경 파일과 빌드 결과만 짧게 보고하기

## 사용자 성향 반영
사용자는 작업 범위가 커지면 오래 고민하고 마감이 늦어질 수 있다.
따라서 Claude는 작업을 크게 벌리지 말고 작은 단위로 끝내는 방향을 제안한다.

새 프로젝트 추가, 전체 리디자인, 트렌드 기능 추가보다 아래를 우선한다.
- 오류 링크 수정
- PDF 버튼 정리
- 대표 프로젝트 선별
- 카드 문장 정리
- 반응형 QA
- 접근성 개선
- PDF 포트폴리오 완성
- 배포 안정화

## 전체 정합성 정리 원칙
- 사용자가 사이트 전체 정합성 정리를 요청한 경우, Home뿐 아니라 실제
  라우트(#/about, #/projects)까지 함께 점검하고 필요한 범위에서 수정할 수
  있다. 위 "사용자 성향 반영"의 "작업을 작게 끝내라"는 규칙은 이런 명시적
  전체 정리 요청에는 적용하지 않는다.
- 전체 리디자인은 금지하지만, 페이지 간 톤과 정보 구조가 다르면 하위
  페이지를 Home 톤에 맞게 재구성할 수 있다.

## 시각/모션 강화 허용 범위
- 전체 리디자인은 금지하지만, 사용자가 명시적으로 첫 화면 시각 강화나
  모션 추가를 요청한 경우 HeroSection, FlowCanvasIllustration,
  ProjectsSection 범위에서는 체감 가능한 시각 변화와 마이크로 애니메이션을
  적용할 수 있다.
- 모션은 장식이 아니라 정보 흐름을 설명하는 용도로 사용한다.
  prefers-reduced-motion 대응 없이 무한 반복 애니메이션을 넣지 않는다.
- 외부 레퍼런스는 복제하지 않되, 레이아웃 밀도, 시선 흐름, 섹션 리듬,
  카드 위계, 모션 방향은 참고할 수 있다.
- 단순히 기존 지시와 일치한다고 판단하고 유지하지 말고, 사용자가 시각
  변화 부족을 지적한 경우 실제 화면에서 보이는 개선을 우선한다.
- 사용자가 Hero 첫 화면의 심심함을 지적한 경우, 전체 리디자인 금지
  원칙보다 HeroSection/FlowCanvasIllustration 범위의 체감 가능한 시각
  강화가 우선된다.
- 컨퍼런스 레퍼런스(if(kakao), FEConf, Figma Config 등)는 복제하지
  않되, 첫 화면의 시선 집중, 중심 오브젝트, 타이포그래피 위계, 섹션
  리듬, 모션 구조는 적극 참고한다.
- 기존 구조와 지시가 이미 맞는다고 판단하고 유지하지 말고, 사용자가
  시각 변화 부족을 지적한 경우 실제 화면에서 보이는 차이를 만들어야
  한다.
- 모션은 장식이 아니라 정보 흐름을 설명하는 용도로 사용한다. 무한
  반복 애니메이션은 금지하고 prefers-reduced-motion 대응을 유지한다.

## 코드 작업 규칙
- 수정 전 변경 계획을 먼저 제안한다.
- 한 번에 너무 많은 범위를 수정하지 않는다.
- 새 라이브러리는 사용자가 명시적으로 요청하지 않으면 추가하지 않는다.
- GitHub Pages 배포 설정은 함부로 변경하지 않는다.
- Supabase 키, env 파일, 비밀 정보는 절대 노출하지 않는다.
- 파일 삭제 전 참조 여부를 확인한다.
- import 경로가 깨지지 않게 확인한다.

## 파일 정리 규칙
- 미사용 이미지, 중복 컴포넌트, 백업 파일, placeholder 데이터는 삭제 후보로 분류한다.
- 삭제 전 "파일 경로 / 삭제 근거 / 참조 여부 / 위험도 / 권장 조치"를 표로 보고한다.
- 확실하지 않으면 삭제하지 말고 TODO로 남긴다.
- node_modules, dist, .cache, .vite는 분석 대상에서 제외한다.

## 검증 규칙
수정 후 가능한 경우 아래를 확인한다.

- npm run build
- 콘솔 에러 없음
- 390px, 430px, 768px, 820px, 1024px, 1366px, 1440px 반응형
- 버튼 겹침 없음
- 가로 스크롤 없음
- 주요 링크 정상
- PDF 버튼은 실제 PDF가 있을 때만 노출
- GitHub, Live Demo, Prototype 링크 정상
