# 김도한 포트폴리오 작업 규칙

## 목적
이 프로젝트는 김도한의 취업용 웹 포트폴리오다.

목표는 화려한 개인 홈페이지가 아니라, 채용자가 30초 안에 지원 방향, 작업 역량, 대표 프로젝트를 이해할 수 있는 신입 UX/UI·웹퍼블리싱 포트폴리오를 만드는 것이다.

## 지원 방향
- 신입 UX/UI 웹디자이너
- 웹퍼블리셔
- 서비스기획형 프론트엔드 주니어
- AI 활용 UX/UI 포트폴리오
- B2B/업무툴 UI 구현형 주니어

## 핵심 포지셔닝
사용자 흐름과 정보 구조를 정리하고, Figma 설계와 React/HTML/CSS/JavaScript 구현을 연결하는 신입 지원자로 보이게 한다.

순수 그래픽 감각만 강조하지 말고, 정보구조, 사용 흐름, 업무형 화면, 구현 가능성, AI 보조 활용을 함께 보여준다.

## 작업 판단 기준
항상 아래 순서로 판단한다.

1. 취업 포트폴리오 완성도
2. 채용자가 빠르게 이해할 수 있는 구조
3. 실제 사용성
4. 반응형/접근성/링크 신뢰도
5. 최신 UI/UX 흐름 반영
6. 사용자가 지속할 수 있는 작업 범위

화려함보다 명확성, 과한 기능보다 신뢰성, 새 프로젝트보다 기존 프로젝트 완성을 우선한다.

## 디자인 방향
- 다크 네이비, 블루그레이, 화이트, 실버 계열 중심
- 차분하고 신뢰감 있는 B2B/업무툴형 분위기
- 콘텐츠 중심의 미니멀한 레이아웃
- 카드형 UI, 명확한 정보 계층, 충분한 여백
- 모바일/태블릿/PC 반응형 안정성 우선
- 우주/네온/장식 요소는 과하면 줄이고 가독성을 우선한다
- 프로젝트 썸네일, 버튼, 라벨, 카드 높이는 일관성을 우선한다

## 외부 레퍼런스 활용 원칙
- 특정 사이트를 그대로 복제하지 않는다.
- 사용자가 지정한 레퍼런스(예: if(kakao)25, FEConf, i-AWARD Korea, DAN25 등)는
  톤앤매너·정보 구조·카드 구성·여백·문장 흐름 참고 용도로만 활용한다.
- 레이아웃/컴포넌트를 그대로 베끼지 않고, 이 프로젝트의 다크 네이비 기반
  디자인 시스템 안에서 재해석한다.
- 사용자가 명시적으로 요청한 경우, 기존 구조를 유지하는 범위에서 시각적
  위계와 레이아웃 밀도를 조정할 수 있다. 단, 외부 사이트의 코드를
  그대로 추출하거나 복제하지는 않는다.

## 디자인 톤 참고 기준 (getdesign)
`getdesign.md`는 복제용 자료가 아니라 디자인 원칙 참고용이다. 이 프로젝트의
기본 톤은 아래 비율로 여러 레퍼런스를 섞어 재해석한 것이며, 특정 브랜드의
로고·고유 그래픽·고유 문구·레이아웃을 그대로 복제하지 않는다.

- **Vercel (45%)** — 타이포그래피, 버튼, 여백, 카드의 정밀도. 전체 레이아웃
  밀도와 여백 기준의 뼈대.
- **Linear (30%)** — Work Style 카드, Projects 카드, 상태 라벨, 미니멀한
  다크 UI 톤.
- **Figma (15%)** — UX/UI 포인트 컬러, Flow Core 주변 라벨. 전체 화면의
  10~15% 비중으로만 사용하고 그 이상 확대하지 않는다.
- **Mintlify/Claude 계열 (10%)** — About 소개문, 프로젝트 설명, 한계/개선점
  문장의 담백하고 절제된 어조.

## Visual Identity Pass
구조(Hero/About/Projects/Contact 배치, Projects Preview vs List 역할)가
안정된 뒤에는, 그 구조를 유지한 채 시각적 완성도를 다듬는 "Visual
Identity Pass"를 허용한다. 이때도 전체 리디자인은 아니며, 아래 시각
키워드를 기준으로 다듬는다.

**Dohan.K 포트폴리오의 시각 키워드**: 정리된 흐름 / 구현 가능한 구조 /
차분한 다크 톤 / cyan-blue 포인트 / 큰 프로젝트 썸네일 / 짧고 명확한 문장.

- 콘텐츠가 "서류/양식"처럼 보이지 않게 한다. 라벨마다 상단 구분선을
  반복해서 긋는 것보다 여백과 문단 흐름으로 구획을 나누는 쪽을
  우선한다.
- AI 그래픽/로봇/회로 이미지, 과한 네온, 우주/파티클은 금지한다.
  움직이는 텍스트도 금지한다.
- 배경 모션은 Flow Stream, 옅은 grid, 카드 reveal, hover 정도까지만
  허용한다.
- AI는 보조 도구로만 표현하고, 태그/라벨을 반복해서 여러 곳에 넣어
  과강조하지 않는다.

## Dohan.K Flow Motion (최종 정의)
Hero에서 시작한 흐름이 About timeline → Projects stage → Footer brand ending으로
이어지는 배경 모션 시스템의 최종 정의다. 아래 요소는 새로 발명하지 않고 이
정의를 기준으로만 다듬는다. 2025~2026 참고 방향은 정돈감보다 개인성·장면감·
진정성·구현 흐름을 우선한다. 17회차는 "강하게 해보고 과하면 줄이는" 실험형
마감(A-3 강한 연출)을 1회 명시적으로 허용해 배경 강도를 기존 대비 25~40%
추가로 올렸다 — 이후 라운드에서 과하다는 피드백이 오면 이 라운드의 수치를
기준으로 낮춘다.

- **정리→설계→구현→검증**: About의 가로 timeline(node+번호+라벨+1줄 설명,
  intro 카드와 역량 카드 사이 독립 블록)이 이 흐름의 canonical 표현이고,
  아래 4개 역량 카드는 같은 흐름의 상세 설명이다. 두 자리 모두 같은 순서·
  같은 단어를 쓴다.
- **D Mark**: `LogoSymbol`(원형 그라디언트 D)은 Hero Flow Core, Navbar,
  Footer Brand Ending 전용 "실물" 마크다. `DMark`(옅은 원 + 큰 D 글자
  워터마크)는 About/Projects 배경에 아주 낮은 opacity로 반복되는 workmark다.
  두 종류를 섞어 쓰지 않고, 한 섹션에 2개 이상 넣지 않는다. Footer는 실물
  로고만 쓰고 DMark를 겹쳐 넣지 않는다(로고 중복 방지).
- **Flow Path**: 섹션 하단의 옅은 세로 그라디언트 커넥터(Hero 하단 패턴)와,
  다음 섹션 상단의 "받는" radial glow(라인이 번져 들어가는 것처럼 보이게
  하는 소프트 확산)를 짝지어 Hero→About→Projects→Contact가 한 흐름으로
  이어지는 느낌을 준다. 무한 반복 애니메이션 없이 정적으로만 사용하고,
  하드 라인 하나만으로 경계를 긋지 않는다(섹션 경계가 선처럼 끊겨 보이면
  실패로 간주한다). Timeline connector는 node 내부를 지나가 보이면 안
  된다 — node 반지름만큼 gap을 두고 시작/종료하거나 node 배경을 완전
  불투명하게 처리해 뒤의 선이 절대 비치지 않게 한다. CTA/Button 뒤로
  장식선이 직접 겹치면 안 되며, 버튼 앞뒤로 충분한 여백을 두고 배치한다
  (절대 위치로 섹션 끝에 라인을 고정하기보다, 버튼 블록 다음 정상 흐름에
  배치해 콘텐츠 길이가 바뀌어도 겹치지 않게 한다).
- **Evidence Badge**: `EvidenceBadges` 컴포넌트가 프로젝트 상세 모달 상단에
  표시하는 검증 배지(구현/설계 완료, 한계 명시, 반응형 확인 등). `status`,
  `limitation`, `is_figma_project` 같은 실제 데이터 필드에서만 도출하고,
  검증되지 않은 문구는 넣지 않는다. 카드 좌상단의 "대표 작업" 배지는 별도
  Evidence Badge 패턴으로 카드에 유지한다.
- **Project Stage**: Selected Works 썸네일을 감싸는 상단 chrome bar + window
  dots + 얇은 border + inner shadow 프레임. 이미지 비율은 항상 `contain`을
  유지하고 `cover`로 바꾸지 않는다.
- **Footer Brand Ending**: Footer의 `LogoSymbol` + "Dohan.K" + "Design to Web
  Interface" 조합을 마지막 장면으로 강화한 마감. Dohan.K 텍스트 크기/weight/
  letter-spacing을 Footer 안에서 가장 강한 타이포로 두고, 상단 accent line과
  브랜드 glow로 경계를 분명히 한다.
- **Contact 모션 예외**: Contact CTA 카드는 Flow Motion의 마지막 행동
  지점이므로, 카드 배경에 한해 24~36초 이상의 아주 느린 diagonal drift
  애니메이션을 예외적으로 허용한다(Hero 예외와 동급). opacity는 낮게
  유지하고 텍스트/버튼 위를 직접 지나가 가독성을 해치면 안 되며, 데스크톱
  전용으로만 적용하고 모바일에서는 표시하지 않는다. `index.css`의 전역
  reduced-motion 규칙(`*, *::before, *::after`)이 모든 `animation`에
  자동 적용되므로 별도 처리는 불필요하다.

## Header 내비게이션 최소화
Home에 About Snapshot이 이미 있으므로 헤더 nav에서 "소개"를 별도 항목으로
두지 않는다. 기본 nav는 **프로젝트 / 연락처 / GitHub** 3개로 유지한다(데스크톱
메뉴, 모바일 Drawer 동일). Hero의 "작업 방식 보기" CTA는 nav 항목이 아니라
About 섹션으로의 스크롤 진입점이므로 계속 유지한다. 로고 클릭은 Home
최상단 이동을 유지한다.

## 디자인 톤 참고 기준 (getdesign) 비율 업데이트
getdesign.md는 여전히 복제용이 아니라 원칙 참고용이며, 이 시스템의 기본 톤은
아래 비율로 재해석한다(이전 25/20/20/20/10/5 비율을 대체):

- **Webflow (30%)** — 콘텐츠 중심 섹션 리듬, 카드 여백감, 장면감 있는 섹션 전환.
- **Raycast (25%)** — 다크 UI의 정밀한 대비, 컴팩트한 라벨/배지, 개인 도구 느낌.
- **Vercel (15%)** — 타이포그래피, 버튼, 카드의 정밀도.
- **Linear (15%)** — Work Style/Projects 카드, 상태 라벨, 미니멀한 다크 톤.
- **Resend (10%)** — 담백한 문장 톤, 절제된 CTA.
- **Figma (5%)** — UX/UI 포인트 컬러, Flow Core 주변 라벨. 화면의 5~10%
  비중을 넘지 않는다.

## 2025~2026 UI/UX 반영 기준
최신 트렌드는 무조건 넣는 것이 아니라 취업용 포트폴리오에 도움이 될 때만 적용한다.

우선 반영:
- 접근성
- 명확한 정보 구조
- 반응형 완성도
- 짧고 명확한 문장
- 프로젝트별 문제 정의와 역할
- AI 활용 과정의 투명한 설명
- 작은 마이크로 인터랙션
- 콘텐츠 중심의 bento/card layout

주의:
- 과한 3D 금지
- 과한 glassmorphism 금지
- 과한 네온 효과 금지
- 의미 없는 애니메이션 금지
- 트렌드라는 이유만으로 새 기능 추가 금지
- 시각디자인,기능 등이 너무 ai티가 나지 말것

## 문장 규칙
- 거짓말 금지
- 과장 금지
- 억지 칭찬 금지
- 검증되지 않은 성과 수치 금지
- 실제 API가 없으면 "데모", "프로토타입", "예시 데이터 기반"으로 표시
- "실무 수준", "완벽", "효율 00% 향상", "사용자 만족도 향상" 같은 검증 안 된 표현 금지
- 부족한 점과 한계를 숨기지 말고 담백하게 표현한다

## AI 활용 표현 규칙
AI는 결과물을 대신 만든 도구가 아니라 보조 도구로 표현한다.

좋은 표현:
- Claude/ChatGPT를 활용해 문장 정리, 코드 점검, 개선안 도출을 보조받았다.
- AI 제안을 그대로 사용하지 않고 요구사항에 맞게 수정하고 검토했다.
- 사용자가 요구사항을 정의하고 최종 판단과 수정 방향을 결정했다.

나쁜 표현:
- AI가 전체 서비스를 제작했다.
- AI로 실무 수준 서비스를 완성했다.
- AI가 사용자 데이터를 분석했다.
- AI가 자동으로 디자인을 완성했다.

## 프로젝트 관리 원칙
프로젝트 리스트는 고정하지 않는다.
대표 프로젝트와 Archive 프로젝트는 사용자의 최신 지시에 따른다.

다만 기본 원칙은 다음과 같다.
- 모든 프로젝트를 메인에 올리지 않는다.
- 대표 프로젝트는 3~4개만 강하게 보여준다.
- 미완성 프로젝트는 대표로 올리지 않는다.
- 실제 API가 없는 프로젝트는 한계를 표시한다.
- Figma 프로젝트에는 GitHub 버튼을 넣지 않는다.
- 오류 링크가 있는 프로젝트는 수정 전까지 대표로 노출하지 않는다.
- 새 프로젝트 추가보다 기존 프로젝트 정리와 마감을 우선한다.

## 답변 방식
사용자는 객관적이고 정확한 피드백을 선호한다.

따라서:
- 불필요한 칭찬 금지
- 막연한 응원 금지
- 장황한 배경 설명 금지
- 부족한 점은 솔직하게 말하기
- 실행 가능한 작업으로 정리하기
- 모르는 것은 추측하지 말고 [확인 필요]로 표시하기
- 수정 전에는 계획을 먼저 제안하기
- 수정 후에는 변경 파일과 빌드 결과만 짧게 보고하기

## 사용자 성향 반영
사용자는 작업 범위가 커지면 오래 고민하고 마감이 늦어질 수 있다.
따라서 Claude는 작업을 크게 벌리지 말고 작은 단위로 끝내는 방향을 제안한다.

새 프로젝트 추가, 전체 리디자인, 트렌드 기능 추가보다 아래를 우선한다.
- 오류 링크 수정
- PDF 버튼 정리
- 대표 프로젝트 선별
- 카드 문장 정리
- 반응형 QA
- 접근성 개선
- PDF 포트폴리오 완성
- 배포 안정화

## 전체 정합성 정리 원칙
- 사용자가 사이트 전체 정합성 정리를 요청한 경우, Home뿐 아니라 실제
  라우트(#/about, #/projects)까지 함께 점검하고 필요한 범위에서 수정할 수
  있다. 위 "사용자 성향 반영"의 "작업을 작게 끝내라"는 규칙은 이런 명시적
  전체 정리 요청에는 적용하지 않는다.
- 전체 리디자인은 금지하지만, 페이지 간 톤과 정보 구조가 다르면 하위
  페이지를 Home 톤에 맞게 재구성할 수 있다.

## 시각/모션 강화 허용 범위
- 전체 리디자인은 금지하지만, 사용자가 명시적으로 첫 화면 시각 강화나
  모션 추가를 요청한 경우 HeroSection, FlowCanvasIllustration,
  ProjectsSection 범위에서는 체감 가능한 시각 변화와 마이크로 애니메이션을
  적용할 수 있다.
- 모션은 장식이 아니라 정보 흐름을 설명하는 용도로 사용한다.
  prefers-reduced-motion 대응 없이 무한 반복 애니메이션을 넣지 않는다.
- 외부 레퍼런스는 복제하지 않되, 레이아웃 밀도, 시선 흐름, 섹션 리듬,
  카드 위계, 모션 방향은 참고할 수 있다.
- 단순히 기존 지시와 일치한다고 판단하고 유지하지 말고, 사용자가 시각
  변화 부족을 지적한 경우 실제 화면에서 보이는 개선을 우선한다.
- 사용자가 Hero 첫 화면의 심심함을 지적한 경우, 전체 리디자인 금지
  원칙보다 HeroSection/FlowCanvasIllustration 범위의 체감 가능한 시각
  강화가 우선된다. 사용자가 "01 WORK STYLE" 등 Hero 인접 섹션의
  평면적임을 지적한 경우에도 동일하게 AboutSection 범위의 시각 강화가
  전체 리디자인 금지보다 우선된다.
- 강한 무한 반복 애니메이션은 금지하지만, Hero 섹션 한정으로
  prefers-reduced-motion에서 완전히 제거되는 조건 하에 12~18초 주기의
  아주 약한 ambient motion(예: connector line 위 highlight 이동,
  spotlight opacity의 미세한 호흡, status strip border의 은은한
  shimmer)은 허용한다. Hero 밖에서는 무한 반복 모션을 넣지 않는다.
- 컨퍼런스 레퍼런스(if(kakao), FEConf, Figma Config 등)는 복제하지
  않되, 첫 화면의 시선 집중, 중심 오브젝트, 타이포그래피 위계, 섹션
  리듬, 모션 구조는 적극 참고한다.
- 기존 구조와 지시가 이미 맞는다고 판단하고 유지하지 말고, 사용자가
  시각 변화 부족을 지적한 경우 실제 화면에서 보이는 차이를 만들어야
  한다.
- 모션은 장식이 아니라 정보 흐름을 설명하는 용도로 사용한다. 무한
  반복 애니메이션은 금지하고 prefers-reduced-motion 대응을 유지한다.
- 컨퍼런스 레퍼런스는 큰 타이포그래피, 중심 오브젝트, 카드 위계도
  함께 참고할 수 있다. 다만 레이아웃/컴포넌트를 그대로 베끼지 않는다.
- "과한 3D/glassmorphism/네온 금지" 원칙은 Hero 섹션 한정으로는
  강한 타이포그래피, glass board(약한 유리질 패널), grid, spotlight,
  line draw, stagger motion(단계별 순차 등장)에는 적용하지 않는다.
  이 원칙은 Hero 이외 섹션(About/Projects/Contact 등)에는 그대로
  적용된다.
- 사용자가 Hero의 "대표성 부족"을 지적한 경우, 전체 페이지 리디자인
  금지 원칙과 별개로 HeroSection/FlowCanvasIllustration **범위 내에서
  보드의 내부 구조 자체를 교체하는 부분 리디자인**을 허용한다. 단,
  Hero 안에 표시하는 프로젝트 데이터를 쓸 경우 Projects 섹션과 동일한
  소스(`data/projectsData.js`)를 재사용해 정합성을 유지한다.
- 과한 3D, 네온, 우주/행성/궤도, 파티클, AI 로봇/회로 감성 연출은
  Hero를 포함한 모든 섹션에서 금지한다(이 항목은 예외 없음).
- **레퍼런스 적용 위치는 섹션별로 구분한다.** 사용자가 레퍼런스별
  적용 위치를 명시한 경우 그 배치를 따른다:
  - Hero: FEConf류 — 큰 타이포그래피, 하나의 중심 오브젝트, 느리고
    약한 ambient motion, 첫 화면 시선 집중.
  - Projects("02 SELECTED PROJECTS"): i-AWARD류 — 모니터/화면 프레임
    안에 대표 작업물(프로젝트) 리스트가 보이는 프리뷰 구조.
  - 전체 톤: DAN25류 — 큰 메시지, 짧은 카피, 과장 없는 테크 브랜드
    안정감.
  같은 레퍼런스 구조(예: 모니터형 프리뷰)라도 사용자가 지정한 섹션이
  아니면 다른 섹션에 넣지 않는다. 잘못된 위치에 이미 적용된 경우,
  전체 리디자인 없이 해당 부분만 올바른 섹션으로 이동/재구성한다.
- 전체 리디자인은 금지하지만, 사용자가 명시한 범위(예: Hero 여백과
  Flow Core 강조, Work Style 배경 리듬, Projects Preview 내부 구조,
  Contact/Footer 마무리 인상)에서는 해당 섹션 내부의 구조 조정을
  허용한다.
- AI는 결과물의 주인공이 아니라 보조 도구로만 표현한다. AI 관련
  문구는 Hero 설명에서 1회, About/Work Style 보조 문장에서 1회
  이내로 제한하고, 그 외 섹션에서 AI를 강조하는 문구를 새로 추가하지
  않는다.
- 배경 장식 효과는 우주/은하/별/궤도 이미지가 아니라 "Flow Stream"
  (얇은 대각선/곡선 라이트 리본, opacity 0.06~0.18, blur 24~36px,
  18~28초의 느린 이동)으로 표현한다. 섹션마다 강도는 다를 수 있고
  Hero 밖에서는 정적으로만(무한 반복 없이) 사용한다. 별/입자/은하
  텍스처는 넣지 않는다.
- 텍스트 자체가 계속 움직이는 애니메이션은 금지한다(진입 시 1회
  fade/translate는 허용). 반복되는 ambient 모션은 배경 라인,
  spotlight, 카드 reveal, hover 효과에만 사용한다.
- Projects Preview는 "현재 선택된 프로젝트 1개의 상세 정보(이름/
  유형/문제/구현 범위/한계) + 오른쪽 큰 모니터 썸네일" 구조를
  원칙으로 한다. 프로젝트 전환 컨트롤(01/03 카운터, Prev/Next,
  작은 dot 등)은 모니터 쪽에 작게 배치하고, 왼쪽 설명 영역에 3개
  프로젝트 정보를 동시에 펼쳐 보여주지 않는다. Preview 아래의 상세
  카드 3개는 별도 영역으로 시각적으로 구분한다(여백 또는 divider +
  제목).
- 사용자가 "밋밋하다/체감이 약하다"고 지적하면, 5~10% 같은 미세
  조정이 아니라 실제 화면에서 분명히 차이가 보이는 수준으로
  개선한다. 이전에 적용한 미세 조정이 충분하다고 판단하지 말고,
  크기/대비/여백을 사용자가 체감할 수 있는 폭으로 다시 조정한다.
- 시각적으로 작은 컨트롤(인디케이터 점 등)도 실제 클릭/터치 hit
  area는 44×44px 이상을 확보한다. 시각적 크기는 유지하되 버튼
  영역을 패딩 등으로 넓힌다.
- Hero CTA 버튼 그룹의 진입 모션(entry motion)은 필수 항목이다.
  부모 Stack/Box에만 opacity·animation을 걸면 버튼 각각의 computed
  style에는 반영되지 않아 "모션 없음"으로 보일 수 있으므로, 각
  버튼(Button) 요소 자체에 opacity 0→1, translateY(10~14px)→0
  애니메이션을 직접 적용한다.
- Flow Core의 orbit 링은 라벨(Figma/React·MUI/Responsive QA/AI
  Assist) 위치에 실제로 선이 끊긴 상태를 유지해야 한다. 라벨 뒤로
  링 라인이 다시 보이면 회귀(regression)로 간주하고 즉시 고친다.
- Hero/Flow Core 크기·구조, Projects Preview("선택 1개 + 모니터
  썸네일") 구조는 사용자가 명시적으로 재변경을 요청하지 않는 한
  그대로 유지한다. 이 두 구조는 여러 라운드에 걸쳐 자리 잡은
  결과이므로 사소한 이유로 다시 바꾸지 않는다.

## Home 중심 구조 원칙
- Home 한 화면 흐름 안에서 지원자 소개, 작업 기준, 대표 프로젝트를 모두
  이해할 수 있어야 한다. `/about` 같은 별도 페이지가 있더라도 Home에는
  About Snapshot(소개/지원 방향/작업 기준/사용 도구/한계 요약)이 반드시
  있어야 하며, "자세한 내용은 별도 페이지에" 식으로 Home을 비워두지
  않는다.
- Projects Preview와 Projects List(카드)는 역할이 다르다. Preview는
  대표 프로젝트를 빠르게 훑어보는 쇼케이스이고, List는 각 프로젝트의
  상세(문제/역할/구현 범위/한계)로 들어가는 진입 카드다. List 카드에
  Preview와 동일한 긴 설명을 반복하지 말고, 상세 설명은 모달/상세
  페이지에서 보여준다.

## 페이지 접근성(네비게이션) 원칙
- Home에 해당 콘텐츠의 Snapshot이 이미 있는 섹션(예: About)은 헤더
  nav에서 별도 페이지(`/about`)가 아니라 **Home 내 스크롤 앵커**로
  연결하는 것을 기본으로 한다. 별도 페이지 링크를 상단에 유지할지는
  사용자의 명시적 선호를 따른다.
- `/about`처럼 별도 라우트로 존재하는 페이지가 nav의 핵심 항목에서
  빠지더라도, 페이지 자체는 삭제하지 않고 유지할 수 있다(직접 URL
  접근·검색엔진 노출 등 다른 경로로 여전히 유효하기 때문).
- 네비게이션에 항목을 추가/변경할 때는 기존 항목(프로젝트/연락처 등)의
  동작과 톤(active 색상, hover, 터치 영역 44×44px 이상)을 그대로
  따른다.

## 프로젝트 설명 문구 정합성 원칙
- 프로젝트 카드 요약, 상세 모달, `/projects` 카드, Projects Preview
  등 같은 프로젝트를 설명하는 여러 위치의 문구는 서로 모순되지
  않아야 한다. 특히 "한계/구현 범위" 문구가 위치마다 다르게 적혀
  있으면 안 되며, 실제 구현 범위(Supabase Auth/DB 연동 여부 등)를
  코드로 직접 확인한 뒤 하나의 사실 기준으로 통일한다.
- 구현 여부가 불확실하면 "구현했다"고 단정하지 말고 "구조 설계",
  "데모", "제한적으로 구현" 등으로 정확히 표현한다.

## cyan 사용 원칙
- cyan(`primary.main`)은 H1 강조, 주요 CTA 버튼, active 상태,
  Flow Core 중심부(개인 브랜드 마크)에만 사용한다.
- 보조 라벨, 카드 eyebrow(카테고리 라벨 등), 반복되는 패널 테두리,
  작은 장식 아이콘, 과한 glow에는 cyan을 남발하지 않는다. 이런
  자리에는 중립 톤(slate/gray 계열)을 기본으로 하고, 강조가 꼭
  필요하면 프로젝트별 accentColor 등 문맥에 맞는 색을 쓴다.
- 색이 있는 텍스트 라벨은 배경 대비 최소 4.5:1을 목표로 한다.
  대비가 애매하면 텍스트 자체를 색으로 칠하기보다 옅은 배경 pill +
  기본 텍스트 색(`text.primary`) 조합으로 표현해 대비 문제를
  구조적으로 없앤다.

## 코드 작업 규칙
- 수정 전 변경 계획을 먼저 제안한다.
- 한 번에 너무 많은 범위를 수정하지 않는다.
- 새 라이브러리는 사용자가 명시적으로 요청하지 않으면 추가하지 않는다.
- GitHub Pages 배포 설정은 함부로 변경하지 않는다.
- Supabase 키, env 파일, 비밀 정보는 절대 노출하지 않는다.
- 파일 삭제 전 참조 여부를 확인한다.
- import 경로가 깨지지 않게 확인한다.

## 파일 정리 규칙
- 미사용 이미지, 중복 컴포넌트, 백업 파일, placeholder 데이터는 삭제 후보로 분류한다.
- 삭제 전 "파일 경로 / 삭제 근거 / 참조 여부 / 위험도 / 권장 조치"를 표로 보고한다.
- 확실하지 않으면 삭제하지 말고 TODO로 남긴다.
- node_modules, dist, .cache, .vite는 분석 대상에서 제외한다.
